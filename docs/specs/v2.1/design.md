# v2.1 设计：AI Studio

> 状态：已确认（2026-07-18，作者完成书面 review）。
> 同迭代：[需求](./requirements.md) · [任务](./tasks.md)。

## 1. 核心决策

### D1：AI 只提出变更，编辑状态仍由单一 Store 拥有

后端 `chat_step` 接收当前 analysis、用户消息和必要上下文，返回经过校验的变更与解释。React 的编辑
Store 是当前编辑状态的唯一拥有者，手调、模板和 AI 都通过同一个应用入口更新它。后端不得维护一份
互相竞争的“AI 当前参数”。

### D2：建议先进入内存中的临时分支

AI 返回后，Store 从当前正式版本派生 `pendingPreview`。画布和面板显示该状态，但数据库当前版本不变。
保留时一次事务写入正式版本；放弃时丢弃内存对象；继续手调时先接受临时基线，再记录后续编辑。

### D3：标量 delta 与曲线整组操作分开

- 标量参数使用相对变化或明确赋值，路径和范围来自统一参数契约。
- 主明度曲线使用“整组控制点替换”，不把单点移动伪装成普通数值 delta。
- 曲线验证失败时整组拒绝，不部分落地；首版不支持 RGB 分通道曲线。

### D4：反思改为用户触发的有界精修

普通消息执行一个 `chat_step`。只有用户点击“AI 精修”才运行最多两轮额外的“查看当前代理图—建议修正—
重新渲染”。每轮都可取消；无有效变化、模型判断完成或达到上限即停止。该选择控制延迟、费用和不可预期漂移。

### D5：正式状态持久化，临时状态不持久化

在应用数据目录建立 SQLite 数据库，v2.1 只包含会话、消息、版本和当前指针的最小表。数据库使用事务、
外键和 `schema_version`；迁移前轮换保留最近三份数据库备份。照片原文件、导出图和缩略图不进入备份。

### D6：外部模型使用去元数据代理图

图像最长边限制为 2048px，统一重新编码且不复制 EXIF。可选的安全拍摄信息由程序解析后作为独立结构化
文本发送，默认白名单为 ISO、快门、光圈、焦距、曝光补偿、白平衡和色彩空间。界面在发送前显示供应商、
代理图数量和元数据开关。

## 2. 状态模型

```text
正式版本 currentVersion
        │
        ├─ 手调/模板 ───────────────→ 新正式版本或同一编辑事务
        │
        └─ AI 建议 → pendingPreview ─┬→ 保留 → 新正式版本
                                     ├→ 继续手调 → 接受基线并继续编辑
                                     └→ 撤销/放弃 → currentVersion
```

`pendingPreview` 至少包含基线版本 ID、候选 analysis、变更摘要、来源消息、代理渲染请求序号和创建时间。
它不是数据库实体。切换照片和关闭窗口必须先解决该状态。

## 3. 参数变更契约

AI 响应分为：

1. 标量操作：参数路径、操作语义、值、理由。
2. 主明度曲线操作：完整控制点列表、理由。
3. 能力说明：无法执行时的限制说明和手动操作步骤。

后端按参数契约白名单解析并规范化；未知操作不透传给前端。规范化后的结果还需经过现有 analysis 校验，
随后才能渲染。若没有任何合法变更，返回说明而不是制造空版本。

## 4. 调用与渲染流程

```text
用户消息
  → 前端提交当前 analysis、最近必要消息、代理图与隐私选项
  → provider 返回结构化建议
  → 后端校验/规范化
  → 前端生成 pendingPreview
  → 防抖渲染代理图
  → 用户保留、撤销、手调或主动 AI 精修
```

预览请求支持取消并带单调请求序号；旧响应晚到时直接丢弃，避免画布回跳。provider 保持无状态，对话历史由
调用方显式提供，因此 Anthropic、OpenAI 兼容、Ollama 与本地 CLI 可复用同一业务流程。

## 5. 持久化边界

最小关系：

- `edit_sessions`：会话、照片引用、最后活动时间。
- `messages`：角色、正文、供应商、状态、所属会话。
- `edit_versions`：父版本、analysis、来源、摘要、创建时间。
- `session_current_versions`：每张会话照片的当前正式版本指针。

正式消息与版本在同一业务事务内提交。失败或取消的模型调用可记录消息状态，但不能移动当前版本指针。
v2.3 再扩展索引根、照片资产、标签和导出记录，v2.1 不提前实现完整图库数据库。

## 6. 能力降级

系统给模型提供当前可用参数清单。当请求超出能力时，结果必须包含：当前不能自动完成的部分、可达到的
近似方案，以及用户在右侧面板中的操作位置。比如局部提亮可建议先做有限的全局阴影调整，并说明真正局部
蒙版将在后续版本提供。不得把不支持的动作翻译成无关参数。

## 7. 异常与恢复

| 场景 | 行为 |
|---|---|
| provider 失败或超时 | 正式版本不变，保留用户消息并允许重试或手调 |
| 用户取消 AI 精修 | 终止后续调用，保留最后可用临时预览 |
| 非法 delta/曲线 | 拒绝非法部分；无合法变更时只显示解释 |
| 渲染失败 | 不展示候选、不写版本，回到正式画面 |
| 数据库提交失败 | 回滚消息与版本事务，临时预览仍可让用户重试保存 |
| 数据库损坏 | 只读恢复并引导从轮换备份恢复，不覆盖原库 |
| 应用崩溃 | 重启恢复最后正式版本，临时预览丢弃 |

## 8. 风险控制

- AI 参数方向不稳定：强制结构化契约、范围校验、mock 确定性测试和真实照片人工验收。
- 对话过长：只发送完成当前任务所需的最近上下文，完整历史仍本地保存。
- SQLite 范围膨胀：v2.1 只实现会话最小表，图库实体延后。
- 隐私误传：代理重新编码与元数据白名单在 provider 适配层之前统一执行，provider 无法绕过。

## 9. 完成条件

需求、状态所有权、持久化和隐私边界互相一致即停止设计扩写；实现细节由后续 TDD 计划确定。
